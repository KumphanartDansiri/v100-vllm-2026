# Chapter 1 — V100 in 2026: dual-engine (vLLM 0.19 + 0.21) source builds + 7 modern model families

> **Status: DRAFT prose** — the benchmark tables render from the frozen SSOT (`data/benchmark_matrix.csv` @ tag `fp8-v100-2026-matrix`); the prose is still settling.

Tesla V100s are everywhere on the surplus market and treated as e-waste by modern inference stacks.
They are not e-waste. With one build trick you can run current Qwen, Gemma, and GLM models — dense
and MoE, FP16 and FP8 — on 8×V100-32GB with vLLM 0.19 and 0.21 today.

## The build trick (and the honest version of "supported")

It would be wrong to say "vLLM supports V100." The accurate statement:

> **vLLM's prebuilt pip wheels dropped practical V100 support at 0.21** — they're built on CUDA ≥12.8,
> whose arch list starts at sm_75. **But vLLM's *source* still has an sm_70 path**: its CMake
> `<12.8` branch still lists `7.0`. So **building from source on a CUDA 12.6 toolchain re-enables
> Volta with no source patch.**

That's the whole unlock, and it applies to both engines we run. You don't get it from
`pip install vllm`; you get it from a source build on the 12.6 toolchain — images
`vllm-v100:vllm021-cu126` and `vllm-v100-py312:vllm019-cu126` (base 0.19 is cu126; Gemma-4 and
GLM-4.7 use a transformers-5 / cu128 variant of the 0.19 image).

We carry **both 0.19 and 0.21 on purpose**, because they win different things:
- **0.21** — newest model support (the latest Qwen/Gemma/GLM architectures land here first), and
  fewer compatibility gaps.
- **0.19** — frequently *faster* on decode (often 10–20%, sometimes more — see the tables) and it
  carries sm_70 more broadly. Far from "stranded," it's the better engine for throughput on most models.

The companion **fp8-w8a16-sm70** plugin — the V100 FP8 W8A16 path, the FP16-MoE fix (Chapter 2), the
coalesced decode kernels, and the FlashAttention/MLA hooks — runs on *both* engines, so every FP8 row
below exists on each, modulo a couple of compatibility gaps noted in the tables.

## Flags that matter on V100
- `--dtype float16` (V100 has no bf16 tensor-core path worth using; fp16 is home turf).
- `--compilation-config '{"mode":0,"cudagraph_mode":"FULL_DECODE_ONLY"}'` — cudagraph is the real
  serving baseline (Chapter 3); it works on Volta and is a 3–5× lift over eager.
- `--skip-mm-profiling` for text-only serving — multimodal models (Gemma, Qwen-VL) otherwise run a
  giant dummy-image profile at startup; skipping it cuts cold start ~45× with zero decode impact.
- TRITON_ATTN is auto-selected on sm_70.

## Reading the results

A few patterns hold across the tables:

- **0.19 usually wins decode.** Same model, same precision, 0.19 is faster than 0.21 on almost every
  row — Qwen3.6-35B-A3B FP8 89.9 vs 74.9 tok/s, Qwen3.6-27B FP8 54.4 vs 46.1, GLM-4.7-Flash 35.4 vs
  31.0 (GLM-4.5-Air is the lone near-tie, 0.21 marginally ahead). If throughput is the goal, 0.19 is
  the engine.
- **0.21 earns its place on compatibility and prefill.** It's where the newest architectures land and
  it has fewer gaps — gemma-4-26B FP8 runs on 0.21 but hits a stock `gemma4.py` MoE error on 0.19 — and
  on some models its prefill is better (GLM-4.7-Flash cold TTFT 143s vs 206s).
- **FP8 is the strongest V100 case, and it's MoE that shines.** At the same TP, FP8 beats FP16 on
  single-user decode for every same-model FP16/FP8 pair measured (q27b 54 vs 40, q35b 90 vs 63). For **MoE** it wins at *every*
  concurrency (q35b FP8 47.7 vs FP16 21.4 tok/s/user at 8 users) — sparse activation keeps decode
  bandwidth-bound, where FP8's half-the-bytes pays off. This is the headline result.
- **Dense models split by concurrency.** FP8 wins the low-user rows (q27b, gemma-4-31B at C1), but
  FP16 takes the 8-user *aggregate*. That C8 deficit is **specific to our current kernel, not to FP8
  in general**: our dequant runs on **CUDA cores**, while FP16 decode reaches cuBLAS **tensor cores**
  that scale better at batch — so dense decode (which streams the whole weight set every token) hits a
  compute wall that a proper **tensor-core / WMMA FP8 decode kernel would close** (future work, no
  timeline). Sparse MoE sidesteps it via per-token sparsity, not because its FP8 is different.
- **TTFT depends on what you measure.** These tables show *cold monolithic* prefill (chunked-off,
  cache-cold). Warm (prefix-cache hit) is sub-second; chunked-prefill inflates cold TTFT on fast
  sparse-MoE models. Never compare a cold number against a warm one — Chapter 3 and the model pages
  keep them separate.

Every model loaded and generated coherent text on V100. "Old but not dead" is a measured claim, not a
slogan.
